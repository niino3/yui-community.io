# マルチテナント アーキテクチャ設計

**バージョン**: 0.1.0  
**作成日**: 2026-03-03  
**ステータス**: ドラフト

> 本システムは特定のコミュニティのためのツールではなく、**パーマカルチャーを実践するあらゆるコミュニティ**が導入できるオープンソースプラットフォームとして設計する。

---

## 1. プロジェクトの位置づけ

### 1.1 ビジョン

```
このシステムは、パーマカルチャーを教えるすべての教室・コミュニティに
無償で提供（オープンソース）するプラットフォームです。

開発者 ──寄付──→ パーマカルチャーコミュニティ全体
```

### 1.2 マルチテナントの意味

「1テナント = 1コミュニティ（1パーマカルチャー教室）」として設計する。

```
yui プラットフォーム
  ├─ テナント A：〇〇パーマカルチャー教室（北海道）
  │    独自トークン: HKD（Hokkaido Community Dollar）
  │    独自Treasury・DAO・メンバー管理
  │
  ├─ テナント B：△△農的暮らしの会（京都）
  │    独自トークン: KYT（Kyoto Token）
  │    独自Treasury・DAO・メンバー管理
  │
  └─ テナント C：□□パーマカルチャー研究所（沖縄）
       独自トークン: OKN（Okinawa Nature Token）
       独自Treasury・DAO・メンバー管理
```

各テナントは**完全に独立した経済圏**を持つ。トークンは原則としてコミュニティをまたいで流通しない。

## 2. アーキテクチャの選択

### 2.1 スマートコントラクト：Factory Pattern

各コミュニティのスマートコントラクト群を **Factory コントラクト** から一括デプロイする。

```
CommunityFactory.sol
  │
  ├─ createCommunity("〇〇パーマカルチャー", "HKD", config)
  │    ↓ 自動デプロイ
  │    ├─ CommunityToken.sol      （HKD トークン）
  │    ├─ GovernanceToken.sol     （HKD-GOV トークン）
  │    ├─ MembershipSBT.sol
  │    ├─ ContributionSBT.sol
  │    ├─ EarthCareSBT.sol
  │    ├─ Escrow.sol
  │    ├─ Governance.sol
  │    ├─ ReputationScore.sol
  │    └─ Treasury.sol
  │
  └─ CommunityRegistry.sol       ← 全コミュニティの一覧を管理
```

**Factory Pattern を選んだ理由：**

| 比較項目 | Factory Pattern（採用） | 共有コントラクト方式 |
|---------|----------------------|-------------------|
| コミュニティの独立性 | ✅ 完全独立 | ❌ 相互影響あり |
| 障害の分離 | ✅ 1コミュニティの問題が他に波及しない | ❌ 単一障害点 |
| カスタマイズ性 | ✅ コミュニティごとにパラメータ設定可 | △ 限定的 |
| デプロイコスト | △ コミュニティ追加時にガス代がかかる | ✅ 低い |
| 管理のシンプルさ | ✅ 各コミュニティが自律管理 | ❌ 複雑 |

### 2.2 フロントエンド：サブドメイン方式

各コミュニティに独自のサブドメインを付与し、ブランドをカスタマイズできる。

```
[プラットフォームのベースURL]
  https://yui-community.io  （ランディング・コミュニティ一覧）

[各コミュニティのURL]
  https://hokkaido.yui-community.io    （〇〇パーマカルチャー教室）
  https://kyoto.yui-community.io       （△△農的暮らしの会）
  https://okinawa.yui-community.io     （□□パーマカルチャー研究所）

または独自ドメイン設定も可能：
  https://permaculture-kyoto.jp        （コミュニティが独自ドメインを取得）
```

各テナントでカスタマイズできる要素：

| 要素 | 例 |
|------|---|
| コミュニティ名 | 〇〇パーマカルチャー教室 |
| トークン名・シンボル | Hokkaido Community Dollar / HKD |
| ロゴ・カラーテーマ | 独自カラーパレット |
| 発行レートの設定値 | 農作業1時間 = 2 HKD（自由に設定） |
| Demurrage率 | 月0.5%〜2%の範囲で設定 |
| 使用言語 | 日本語 / 英語 / その他 |

### 2.3 バックエンド：スキーマ分離方式

PostgreSQL の **スキーマ分離（Schema per Tenant）** を採用する。

```
PostgreSQL
  ├─ schema: hokkaido
  │    ├─ users
  │    ├─ tasks
  │    ├─ transactions
  │    └─ earth_care_activities
  │
  ├─ schema: kyoto
  │    ├─ users
  │    ├─ tasks
  │    └─ ...
  │
  └─ schema: public （共通・プラットフォーム管理）
       ├─ communities（テナント一覧）
       ├─ contract_registry（各コミュニティのコントラクトアドレス）
       └─ platform_config
```

**スキーマ分離を選んだ理由：**
- テナント間のデータ漏洩リスクが低い
- コミュニティ単位でのデータエクスポート・削除が容易（GDPRへの対応）
- 必要に応じてコミュニティを別のDBに移行できる

## 4. 新規コミュニティのオンボーディングフロー

### 4.1 コミュニティ管理者（パーマカルチャー教室の先生）の操作

```
【ステップ1】プラットフォームサイトでコミュニティを作成

  https://yui-community.io にアクセス
  ↓
  「新しいコミュニティを作成する」
  ↓
  必要情報を入力：
    - コミュニティ名（例：〇〇パーマカルチャー教室）
    - サブドメイン（例：hokkaido）
    - トークン名・シンボル（例：HKD）
    - 所在地域
    - Demurrage率・発行レートの初期設定

【ステップ2】スマートコントラクトの自動デプロイ

  CommunityFactory.sol が実行される
  ↓
  全コントラクトが自動でデプロイされる（ガス代が発生）
  ↓
  CommunityRegistry に登録される

【ステップ3】コミュニティサイトの開設

  https://hokkaido.yui-community.io が利用可能になる
  ↓
  管理者がロゴ・カラー・ルール文書をカスタマイズ
  ↓
  メンバーをLINEやSNSで招待
```

### 4.2 コミュニティ作成のコスト

| 項目 | コスト | 備担 |
|------|-------|------|
| コントラクトデプロイ（ガス代） | 約 $5〜20 相当（Polygon） | コミュニティ管理者が負担 |
| サブドメイン設定 | 無料 | プラットフォームが提供 |
| アプリ利用料 | 無料（オープンソース） | 寄付モデル |
| 独自ドメイン取得 | 任意・自己負担 | |

## 5. クロスコミュニティの設計方針

### 5.1 原則：トークンはコミュニティ内クローズド

各コミュニティのユーティリティトークン（YUI系）は **他コミュニティと交換しない**。これは：
- 法的安全性の維持（クローズドな流通であることが重要）
- 各コミュニティの経済的自律性の保護
- 投機・アービトラージの防止

### 5.2 クロスコミュニティで共有できるもの

| 共有可能なもの | 仕組み |
|-------------|-------|
| **メンバーシップSBT** | 複数コミュニティのSBTを1つのウォレットに持てる（マルチコミュニティ参加） |
| **信頼スコア** | 他コミュニティのContributionSBTを参照した「プラットフォーム全体の信頼スコア」表示（任意） |
| **Earth Care SBT** | 環境貢献実績は全コミュニティで可視化（地球への貢献は共通価値） |
| **ガバナンスの知見** | 成功した提案・ルール設計をテンプレートとして共有 |

### 5.3 将来的な検討：コミュニティ間ブリッジ

長期的には、コミュニティ間での **特定活動の相互承認** を検討できる：

```
例：「パーマカルチャー教室の受講」という Earth Care 活動を
    複数コミュニティが共同で承認する仕組み

Aコミュニティ + Bコミュニティ が共同承認
  → それぞれのコミュニティで Earth Care SBT が発行される
  → トークンの移転は発生しない
```

## 6. オープンソースとしての提供モデル

### 6.1 ライセンス

**MIT License** を採用。商用利用・改変・再配布を自由に許可する。

```
理由：
- パーマカルチャーの「余剰の分かち合い（Fair Share）」原則に合致
- より多くのコミュニティに使ってもらうことが目的
- Fork して独自カスタマイズも歓迎
```

### 6.2 コントリビューション

各コミュニティが独自に開発した機能をプラットフォームに還元できる仕組みを用意する。

```
コミュニティが機能を開発
  ↓
GitHub にPull Request
  ↓
プラットフォームのDAO（または維持管理者）が審査・マージ
  ↓
全コミュニティが恩恵を受ける
```

### 6.3 寄付・サポートモデル

プラットフォームの維持・開発のための資金調達：

| 方法 | 内容 |
|------|------|
| コミュニティからの任意寄付 | 各コミュニティのTreasuryから自発的に開発者へ寄付 |
| グラント申請 | Web3系・農業系・環境系の助成金を活用 |
| スポンサーシップ | パーマカルチャー関連の団体・企業からの支援 |

## 8. 技術的な追加考慮事項

### 8.1 アップグレーダビリティ

オープンソースで長期運用するため、スマートコントラクトの **Upgradeable Pattern（Proxy）** を採用する。

```
OpenZeppelin の TransparentUpgradeableProxy を使用
  → バグ修正・機能追加時にコントラクトを再デプロイ不要
  → 各コミュニティのデータ・残高を保持したままアップグレード可能
```

### 8.2 セルフホスト対応

技術力があるコミュニティは独自サーバーで全システムを運用できる。

```
提供形態：
  A. マネージドサービス（開発者が運用）
     → yui-community.io のサブドメインを使用
     → 設定だけで使える

  B. セルフホスト（コミュニティが自分で運用）
     → Docker Compose で一発起動
     → 独自ドメイン・独自サーバーで完全に自律運用
```
